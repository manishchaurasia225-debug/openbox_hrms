README.md 
1    # OGM HRMS — Enterprise HR Operations Platform
2    
3    A production-grade, single-company internal **HR Operations Platform**: the single source of truth for
4    employee information and HR operations. Backend is feature-complete across 24 modules (auth/RBAC,
5    employees, attendance, leave, documents, payroll, reimbursements, dashboards, reports, automation,
6    email/WhatsApp, AI assistant, global search, audit, and system administration).
7    
8    - **Stack:** Java 21 · Spring Boot 4.1 · Spring Security (JWT) · Spring Data JPA/Hibernate ·
9      PostgreSQL · Flyway · Gradle (Kotlin DSL) · springdoc-openapi (Swagger) · Actuator
10   - **Testing:** JUnit 5 · Testcontainers (real PostgreSQL for integration tests)
11   
12   ---
13   
14   ## TL;DR — start in three commands
15   
16   ```bash
17   cp .env.example .env      # then edit DB_* and SUPER_ADMIN_*
18   # (make sure PostgreSQL is running and the 'hrms' database exists — see below)
19   ./scripts/run-dev.sh
20   ```
21   
22   Then log in with the `SUPER_ADMIN_EMAIL` / `SUPER_ADMIN_PASSWORD` you set in `.env`, and open the API
23   docs at <http://localhost:8080/swagger-ui.html>.

41   ## 2. PostgreSQL setup
42   
43   ### Option A — Docker (recommended)
44   
45   ```bash
46   docker run --name hrms-postgres \
47     -e POSTGRES_DB=hrms \
48     -e POSTGRES_USER=hrms \
49     -e POSTGRES_PASSWORD=hrms \
50     -p 5432:5432 -d postgres:17
51   ```
52   
53   That creates the `hrms` database automatically. Set `DB_USERNAME=hrms` / `DB_PASSWORD=hrms` in your
54   `.env` to match.
55   
56   ### Option B — existing/local PostgreSQL
57   
58   Create the database once:
59   
60   ```bash
61   createdb -h localhost -p 5432 hrms
62   # or:  psql -h localhost -U postgres -c 'CREATE DATABASE hrms;'
63   ```
64   
65   The application does **not** create the database itself — but it **does** create all tables via Flyway
66   migrations on startup (you don't run any DDL by hand).

70   ## 3. Environment setup
71   
72   All configuration comes from environment variables, loaded from a git-ignored `.env` (or `.env.dev`).
73   Copy the template and edit it:
74   
75   ```bash
76   cp .env.example .env
77   ```
78   
79   At minimum set:
80   
81   - `DB_URL`, `DB_USERNAME`, `DB_PASSWORD` — your PostgreSQL connection.
82   - `SUPER_ADMIN_EMAIL`, `SUPER_ADMIN_PASSWORD` — the first admin to bootstrap (see below).
83   
84   Everything else has a sensible default. `.env.example` documents every variable. The dev profile ships
85   an **insecure** JWT secret so the app boots without `HRMS_JWT_SECRET`; always set a real one outside
86   local dev (`openssl rand -base64 48`).
87   
88   > **Do not assign `HRMS_JWT_SECRET=` an empty value.** An empty assignment is exported by the run
89   > scripts and *overrides* the dev fallback with a blank secret, so the app refuses to start. Leave the
90   > line commented out (unset) in dev, or set a real secret. Same applies to any other secret with a
91   > profile default.
92   
93   For a browser SPA on another origin, set `HRMS_CORS_ALLOWED_ORIGINS` (comma-separated). The dev profile
94   already allows `http://localhost:5173` and `http://localhost:3000`; with none configured, no
95   cross-origin access is granted (the safe default).
96   
97   > Secrets live only in `.env*` (git-ignored except `.env.example`) — never commit real credentials.

101  ## 4. First startup & the Super Admin bootstrap
102  
103  The application seeds **no user accounts** by default — only roles and the permission catalogue. On the
104  **first** startup, if **both** `SUPER_ADMIN_EMAIL` and `SUPER_ADMIN_PASSWORD` are set **and** no user
105  with that email exists, a single **Super Admin** is created so you can log in.
106  
107  - **Login is by email** — there is no separate username.
108  - The step is **idempotent**: once the admin exists, these values are ignored on later startups.
109  - The password is stored **BCrypt-hashed**, never in plaintext.
110  - If the values are unset, startup logs *"No hrms.bootstrap.super-admin credentials configured;
111    skipping admin bootstrap"* and you won't be able to log in until a user exists.
112  
113  Start the app:
114  
115  ```bash
116  ./scripts/run-dev.sh
117  ```
118  
119  `run-dev.sh` loads `.env`/`.env.dev`, verifies required variables, checks that PostgreSQL is reachable,
120  prints the Swagger/health URLs, then runs the app. On success you'll see Flyway apply the migrations and
121  a log line like `Bootstrapped Super Admin user 'admin@example.com'` (first run only).
122  
123  ### Log in
124  
125  ```bash
126  curl -X POST http://localhost:8080/api/v1/auth/login \
127    -H 'Content-Type: application/json' \
128    -d '{"email":"admin@example.com","password":"<your SUPER_ADMIN_PASSWORD>"}'
129  ```
130  
131  The response contains `data.accessToken` — send it as `Authorization: Bearer <token>` on subsequent
132  API calls (or paste it into Swagger UI's **Authorize** dialog).

136  ## 5. API docs, health & endpoints
137  
138  | What | URL |
139  |------|-----|
140  | Swagger UI | <http://localhost:8080/swagger-ui.html> |
141  | OpenAPI JSON | <http://localhost:8080/v3/api-docs> |
142  | Health | <http://localhost:8080/actuator/health> |
143  | API base | `http://localhost:8080/api/v1/...` |
144  
145  Swagger UI, the OpenAPI document, and the health endpoint are public; all `/api/v1/**` endpoints (except
146  the auth token endpoints) require a JWT. Every operation carries a summary/description and is grouped by
147  tag in Swagger UI.
148  
149  ### Postman
150  
151  A ready-to-import Postman collection covering **every** endpoint (generated from the live OpenAPI
152  document, so it stays in sync with the API) lives in [`postman/`](postman/):
153  
154  - `postman/hrms.postman_collection.json` — 32 folders, all API operations, collection-level Bearer auth.
155  - `postman/hrms-local.postman_environment.json` — the *HRMS – Local (dev)* environment.
156  
157  Import both, select the environment, then run **Authentication › Login** once — its test script captures
158  the token into the `{{accessToken}}` variable, so every other request is authorized automatically. Set
159  the `adminEmail` / `adminPassword` variables to your bootstrapped Super Admin first.

163  ## 6. Verify the environment
164  
165  With the app running, in another terminal:
166  
167  ```bash
168  ./scripts/verify-dev.sh
169  ```
170  
171  It reports **PASS/FAIL** for: environment variables loaded, database connectivity, Flyway migration
172  status, seeded roles & permissions, Super Admin existence, health endpoint, login endpoint, an
173  authenticated request, and the OpenAPI/Swagger endpoints. Exit code is non-zero if any check fails.

177  ## 7. Running tests
178  
179  ```bash
180  ./gradlew build          # compile + run the full unit + integration suite
181  ./gradlew test           # tests only
182  ```
183  
184  Integration tests use **Testcontainers**, which starts a throwaway PostgreSQL container — so **Docker
185  must be running**. No local database or `.env` is needed for tests (they are fully self-contained).
186  
187  Coverage is reported by **JaCoCo**, generated automatically after the test run:
188  
189  ```bash
190  ./gradlew test jacocoTestReport
191  open build/reports/jacoco/test/html/index.html    # human-readable HTML report
192  ```
193  
194  The suite (98 tests across unit + integration) currently covers ~76% of instructions / ~75% of lines.

198  ## 8. Profiles & production
199  
200  | Profile | Purpose |
201  |---------|---------|
202  | `dev` (default) | `ddl-auto=validate`, SQL logging, insecure fallback JWT secret for convenience. |
203  | `prod` | `ddl-auto=validate`, connection pool tuning, no SQL logging, no stack traces to clients. Requires **all** secrets via the environment (`run-prod.sh` fails fast if missing). |
204  
205  Activate prod with `SPRING_PROFILES_ACTIVE=prod` (or `--spring.profiles.active=prod`) and run
206  `./scripts/run-prod.sh` (builds a bootJar and launches it). In production, supply `HRMS_JWT_SECRET`,
207  `DB_*`, and the `SUPER_ADMIN_*` bootstrap via real environment variables or a secrets manager — never
208  files. See [`.claude/deployment.md`](.claude/deployment.md) for details.
209  
210  > Redis caching and AWS S3 document storage are on the roadmap but **not yet wired** into the running
211  > app; document binaries currently persist to the local filesystem (`HRMS_STORAGE_PATH`), and email is
212  > logged rather than sent unless you configure SMTP.

216  ## 9. Troubleshooting
217  
218  | Symptom | Likely cause / fix |
219  |---------|--------------------|
220  | `run-dev.sh` says *PostgreSQL is NOT reachable* | Start Postgres (section 2) and confirm host/port in `DB_URL`. |
221  | App fails at startup with a Flyway/`schema validation` error | The DB schema is out of sync. For a fresh dev DB, drop and recreate it so Flyway re-applies from V1: `dropdb hrms && createdb hrms`. |
222  | `FlywayValidateException` / checksum mismatch | A migration file changed after being applied. Use a clean database in dev. |
223  | Login returns 401 | Wrong email/password, or no Super Admin was bootstrapped — check the startup log and that `SUPER_ADMIN_*` were set **before first startup**. |
224  | Can't log in on a DB that already has data | The bootstrap only runs when the user doesn't exist. Set `SUPER_ADMIN_*` to an email that isn't present, or create a user via the admin API. |
225  | `HRMS_JWT_SECRET` error outside dev | The secret must be ≥ 32 bytes. Generate: `openssl rand -base64 48`. |
226  | App won't start in dev: *HRMS_JWT_SECRET is not configured* | Your `.env` assigns `HRMS_JWT_SECRET=` (empty), which overrides the dev fallback with a blank secret. Comment the line out (leave it unset) or set a real secret. |
227  | Browser SPA gets a CORS error calling the API | Add the SPA's origin to `HRMS_CORS_ALLOWED_ORIGINS` (comma-separated). Dev already allows `localhost:5173`/`3000`. |
228  | Tests fail to start containers | Docker isn't running, or the daemon isn't reachable by Testcontainers. |
229  | Port 8080 in use | Set `SERVER_PORT` in `.env`. |
